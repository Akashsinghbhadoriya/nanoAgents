# archmate

An AI copilot that turns plain-English product ideas into architecture diagrams, database schemas, API designs, and scaling plans.

## What it does

You describe a product. archmate produces a complete system design.

```
Input:  "Build a food delivery app"

Output:
  - Architecture diagram
  - Database schema
  - API design
  - Scaling considerations
```

## Why

Very few good AI tools exist for architecture design. Most AI assistants give vague, generic answers. archmate goes deep — generating structured, editable, production-aware designs grounded in real engineering trade-offs.

## How it works

archmate uses an agentic loop built from first principles:

```text
User Prompt (product idea)
         ↓
      Router
      ↙    ↘
  direct   plan
              ↓
           Planner
         (decompose design into subtasks)
              ↓
           Executor
         ↙         ↘
  DesignAgent    Reflector
  (ReAct loop)   (review + revise)
      ↓
  Artifacts
  - Architecture diagram (Mermaid / editable)
  - Database schema (SQL / NoSQL)
  - API spec (OpenAPI)
  - Scaling notes
```

Each design artifact is generated by a specialized agent step, reflected upon, and revised until it meets quality criteria.

## Project Structure

```text
archmate/
├── agent_system/
│   └── agent_system.py       # Orchestrates router, planner, executor, memory, logger
├── agents/
│   ├── agent.py              # ReAct loop
│   ├── react_prompt.py       # Prompt builder for ReAct
│   ├── react_parser.py       # Parses LLM output into actions
│   ├── tool_registry.py      # Registers and executes tools
│   ├── state.py              # Agent state (actions + observations)
│   └── step_trace.py         # Per-step execution trace
├── tools/
│   ├── base.py               # Tool base class
│   ├── architecture.py       # Generates architecture diagrams (Mermaid)
│   ├── schema.py             # Generates database schemas
│   ├── api_design.py         # Generates OpenAPI specs
│   ├── scaling.py            # Produces scaling analysis
│   ├── search.py             # Web search for design patterns
│   └── memory.py             # Stores design context and preferences
├── memory/
│   ├── conversation_memory.py  # Short-term conversation history
│   ├── persistent_memory.py    # Long-term memory across sessions
│   └── fact_memory.py          # Persistent key-value facts (JSON)
├── planning/
│   ├── planner.py            # Decomposes design goals into subtasks
│   ├── executor.py           # Runs each design task through the agent
│   ├── prompts.py            # Planner and replanner prompt templates
│   ├── plan.py               # Plan dataclass
│   └── task.py               # Task dataclass
├── routing/
│   ├── router.py             # Routes queries to direct / tool / plan
│   └── routing_prompt.py     # Routing prompt with rules and examples
├── reflection/
│   ├── reflector.py          # Evaluates design quality, triggers revision
│   └── reflection_prompt.py  # Reflection prompt template
├── llm/
│   ├── base.py               # LLM base class
│   └── ollama_llm.py         # Ollama backend (default: qwen3:8b)
├── logger/
│   ├── run_logger.py         # Saves full run traces to JSON
│   └── progress_report.py    # Console progress reporting
├── tests/
├── app.py                    # Wires all components together
└── main.py                   # Entry point
```

```bash
pip install -r requirements.txt

ollama pull qwen3:8b

# Run
python main.py
```

## Example session

```
You: Design a ride-sharing app like Uber

archmate:

## Architecture
[Mermaid diagram — editable]

Services:
- API Gateway
- Rider Service
- Driver Service
- Matching Engine
- Pricing Service
- Notification Service
- Trip Service
- Payment Service

## Database Schema
riders(id, name, phone, rating, created_at)
drivers(id, name, phone, vehicle_id, rating, status)
trips(id, rider_id, driver_id, origin, destination, status, fare)
...

## API Design
POST /trips/request     — Rider requests a trip
GET  /trips/:id         — Get trip status
POST /drivers/location  — Driver updates location
...

## Scaling Considerations
- Matching Engine: geospatial index (PostGIS / Redis GEO)
- Driver location updates: WebSockets + Redis pub/sub
- Pricing: isolated service, cache surge multipliers
- Trip DB: partition by city/region
- ETA estimation: async worker pool
```

## Roadmap

### Phase 1 — Core Design Generation
- [ ] Architecture diagram generation (Mermaid)
- [ ] Database schema generation
- [ ] API spec generation (OpenAPI)
- [ ] Scaling analysis

### Phase 2 — Interactivity
- [ ] Editable diagrams in the browser
- [ ] Follow-up refinement ("make it multi-region")
- [ ] Export to PNG / PDF / Notion

### Phase 3 — Design Intelligence
- [ ] Trade-off analysis (SQL vs NoSQL, monolith vs microservices)
- [ ] Cost estimation
- [ ] Known failure modes and mitigations

### Phase 4 — Multi-Agent Design Review
- [ ] Separate agents for security, performance, reliability review
- [ ] Agents debate design decisions
- [ ] Consensus-driven final design

## Philosophy

Built from first principles — no LangChain, no AutoGen. Each component is small, readable, and educational. The goal is to understand agent primitives deeply, then combine them into a product that solves a real problem.

Learn primitives first. Build systems second. Understand abstractions always.
